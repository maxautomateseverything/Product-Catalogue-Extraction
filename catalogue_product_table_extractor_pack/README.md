# Catalogue Table Extractor — Version 9

Version 9 changes the extraction logic from **table-first formatting** to
**SKU-anchored relationship extraction**.

The primary objective is now:

> Attach the correct attribute value to the correct SKU and the correct source
> header, while leaving uncertain relationships unresolved rather than guessing.

The program still creates a faithful reconstructed table, but product records
are no longer generated by assuming that every table has a conventional header
row followed by one product per row.

---

# 1. What changed

## Matrix selection is based on SKU preservation

PyMuPDF4LLM proposes table regions. Within each region, Version 9 compares:

- The supplied PyMuPDF4LLM matrix.
- A PyMuPDF `lines` grid.
- A PyMuPDF `lines_strict` grid.

Each candidate is scored using:

- Exact SKU registry matches.
- Number of independent SKU cells.
- Number of unique SKUs.
- Table-region overlap.
- Row and column structure.
- Penalties when many SKUs are merged into one cell or header.

This directly addresses the Version 8 failure where a complete column of GEWISS
codes was merged into a header. The ordinary `lines` grid preserves those codes
as individual cells and therefore wins the Version 9 score.

## Every confirmed SKU becomes an anchor

A cell matching the canonical SKU registry is always treated as an SKU anchor,
regardless of whether it occurs:

- On the left.
- On the right.
- In the middle of a matrix.
- Several times on one visual row.
- Across a header band as an SKU-column matrix.

A strong pattern match absent from the registry becomes an unconfirmed anchor
and is sent to review. Fuzzy matching never silently confirms an SKU.

## Attributes are found from the anchor

For each SKU anchor, the program examines:

1. **Upward header path**
   - Product family
   - IP rating
   - Wiring type
   - Other hierarchical column meaning

2. **Same-row attribute columns on the left and right**
   - Rated current
   - Voltage
   - Frequency
   - Poles
   - Reference position
   - Applications
   - Dimensions
   - Other table-specific fields

3. **Merged-cell coverage**
   - A value is repeated only when a real PDF cell geometrically spans the SKU
     row.
   - A genuine empty product slot remains empty.

4. **Adjacent-page continuation relationships**
   - Specifications can be inherited from an adjacent page.
   - Applications and complementary context can flow in the other direction.

The search can cross other SKU columns because the registry identifies those
cells as products rather than attribute values.

## Attribute precision is calculated per relationship

Every candidate relationship has its own confidence:

```text
>= 0.90   automatically accepted
0.70–0.89 retained but flagged
< 0.70    not assigned automatically
```

An uncertain footnote does not invalidate a reliable voltage or current value.

## Original and normalized values are both retained

Each accepted relationship stores:

```text
source_attribute
normalized_attribute
candidate_value / value_raw
value_normalized
unit
source page, table, row and column
source cell bounding box
header path
relationship direction
merged/continuation inheritance
confidence
```

# 2. How a GEWISS matrix is interpreted

For the first row of the supplied two-page example:

```text
16 A | 100 ÷ 130 V | 50/60 Hz | 2P+E | 4 |
GW 60 001 FH | GW 60 001 H | GW 60 023 FH | GW 60 023 H
```

Version 9 creates four independent product records.

For `GW 60 001 FH`:

```text
rated_current = 16 A
rated_voltage = 100 ÷ 130 V
frequency = 50/60 Hz
poles = 2P+E
reference_h = 4
product_type = Straight plugs
ip_rating = IP44/IP54
wiring_type = Fast wiring
applications = Standard use
```

The application is inherited from the automatically joined next page.

For `GW 62 001 FH`, the next page supplies the SKU and local column hierarchy,
while the first page supplies the shared specifications:

```text
rated_current = 16 A
rated_voltage = 100 ÷ 130 V
frequency = 50/60 Hz
poles = 2P+E
reference_h = 4
product_type = Straight plugs
continuation_product_type = Straight connectors
ip_rating = IP44/IP54
wiring_type = Fast wiring
applications = Standard use
```

The complete displayed SKU remains independent. Version 9 does not reduce
`GW 60 005 FH`, `GW 60 005 H`, and `GW 60 005 WH` to a common base SKU.

# 3. Files in the pack

```text
catalogue_table_extractor.py
  Main batching, registry, PyMuPDF4LLM and catalogue aggregation process.

  SKU-anchor, matrix scoring, header paths, attribute relationships and
  continuation joins.

run_extractor.ps1
  Guided Windows launcher.

  Repeatable configuration example.

profiles/
  Optional reusable generic, GEWISS and ROBUS profiles.

examples/
  Tested two-page GEWISS results and diagnostic evidence.
```

The two Python files must remain in the same folder.

# 4. Install or update the pack

## Use a new folder

Extract Version 9 into a new folder, for example:

```text
C:\CatalogueTableExtractorV9
```

Do not copy only the main script. Version 9 requires the new
`dynamic_table_model.py` beside it.

## Install Python packages

Open PowerShell in the new folder and run:

```powershell
py -m pip install --upgrade pymupdf pymupdf4llm pillow
```

Check the command:

```powershell
py ".\catalogue_table_extractor.py" --help
```

# 8. Pass criteria for the two-page test

Confirm these records first:

```text
GW 60 001 FH
GW 60 001 H
GW 60 023 FH
GW 60 023 H
GW 60 005 WH
GW 62 001 FH
```

The pack includes:

```text
examples\GEWISS_18_19_v9_reference_records.csv
```

Compare it with each page's `10_normalized_product_records.csv`.

The following must be true:

- Fast and screw wiring columns remain separate.
- IP44/IP54 and IP66/IP67/IP68/IP69 remain separate.
- No block of SKUs appears inside a header.
- `Standard use` attaches to the correct row group.
- Only `WH (*)` variants receive the phase-inverter footnote.
- Blank product positions do not copy an adjacent SKU.
- Page-two products inherit the correct page-one specifications.

# 12. AI usage

Version 9 does not need AI to format clean registry-backed matrices.

AI should be reserved for:

- Unresolved headers.
- Low-confidence relationships.
- Visual-only colour or icon fields.
- Conflicting continuation evidence.
- Pages already present in `review_queue.csv`.

For the current CPU-only computer, use `qwen3-vl:4b-instruct` for exception
processing and retain the 8B model for a smaller difficult-page set.

---

# 13. Generalization rules

The engine contains no `if manufacturer == GEWISS` or `if manufacturer == ROBUS`
branches.

Optional profiles may provide only reusable information:

- SKU regex patterns.
- Header aliases.
- Unit patterns.
- Known non-SKU option codes.
- Footnote semantic rules.
- Confidence thresholds.

Table geometry, page numbers, SKU positions and values are detected from the PDF
and registry at runtime.

---

# 14. Known scope

This pack extracts tables and table-like matrices.

It does not yet consolidate non-table feature lists, prose, icon strips,
construction paragraphs or technical diagrams. Those belong in the separate
non-table product-data extractor discussed earlier.
